在过去的几周里，Lucas Gerads 一直在尝试使用 Claude Code 进行硬件开发。他观察到一些项目将 Claude 用于电路设计，类似于软件开发中的用法：用自然语言编写提示，让 Claude 生成电路。但他发现，用纯英语描述想要构建的电路有些棘手，对于简单电路尚可，但面对复杂设计时就变得困难。同时，他意识到当 Claude 能获得即时反馈时，其能力才会真正凸显。，因此，他尝试了一种新方法：让 Claude 直接访问他的示波器和 SPICE 仿真器。这为验证 SPICE 电路与模型、嵌入式编程以及数据分析带来了极大价值。特别是数据分析，以往非常繁琐，例如归一化时间轴、对齐数据等，通常只能靠目测。，他通过一个特意简化的演示展示了其工作流程：先进行 SPICE 仿真，然后用示波器进行实际测量，最后使用 Claude Code 进行自动化验证。虽然演示中的电路和微控制器（MCU）很简单，但该方法能很好地扩展到更复杂的电路和真实的嵌入式项目中。，关键经验总结如下：，对于示波器：Claude 无法感知物理连接，因此需明确配置，避免让它猜测连线。确保 Claude 始终获取最新的测量数据，不要将原始数据直接塞入其上下文，而是保存到文件并让 Claude 间接交互。，对于微控制器：需明确提供引脚分配/引脚复用映射图。准备一个 Makefile，暴露 build、flash、ping 和 erase 等功能，并引导 Claude 依赖此文件，而不是动态构建这些命令。，相关工具仓库包括：用于 LeCroy 示波器的 MCP 服务器 lecroy-mcp、封装 spicelib 的 MCP 服务器 spicelib-mcp，以及视频中演示示例的文件 rc-filter-demo-files。该方法的核心价值在于通过集成仿真、测量与 AI 驱动的自动化验证，显著提升了硬件开发与调试的效率和可靠性。

GitHub用户GainSec近日开源了AutoProber项目，这是一个专为硬件黑客设计的自动化飞针探测堆栈。该项目利用胶带、旧摄像头和CNC机床等简易材料，构建了一套由AI驱动的硬件探测机械臂系统，旨在实现代理驱动的目标发现、显微镜映射、安全监控的CNC运动、探针检查以及受控的引脚探测。，AutoProber的核心功能是提供一个完整的自动化工作流，使AI代理能够安全地从识别新目标板逐步执行到对单个引脚进行探测。其工作流程包括：代理接收项目信息并连接所有硬件；确认各部件功能正常后执行归位和校准；安装定制探针和显微镜头；当代理检测到平台上有新目标时，会自动定位目标位置，采集并记录带有XYZ坐标的独立图像帧，同时标注焊盘、引脚、芯片等关键特征；随后将图像帧拼接成标注地图，并通过Web仪表板提交探测目标供用户审核；最终对批准的目标执行探测并反馈结果。，该系统支持通过Web仪表板、Python脚本或AI代理直接控制所有硬件。项目仓库提供了完整的源代码候选版本，包含Python控制代码、仪表板、CAD文件和详细文档，方便用户自行构建AutoProber。，安全模型是项目的关键设计，强调需将其视为机器控制系统而非普通Web应用。安全机制包括：忽略GRBL的Pn:P信号，不将CNC探针引脚作为可信限位开关；独立的安全限位信号从示波器通道4读取，并在所有运动期间持续监控；任何通道4触发、电压异常、CNC报警或实际X/Y/Z限位引脚触发均会立即停止运动，且代理或操作员必须手动停止并报告，系统不提供自动恢复运动。开发者强烈建议用户在操作硬件前仔细阅读docs/safety.md和docs/operations.md安全文档。，该项目展示了如何利用低成本材料和开源软件栈，结合AI代理与自动化控制，为硬件安全研究领域提供了一种创新的物理探测解决方案。
