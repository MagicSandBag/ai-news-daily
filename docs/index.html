Vercel插件在Claude Code中请求读取用户所有提示词引发隐私担忧。该插件旨在协助部署、提供框架指导和技能注入，但其内置的框架检测功能并未用于限制遥测数据收集范围。问题核心在于，即使用户项目与Vercel无关（无vercel.json配置文件、无Next.js配置、无Vercel依赖），插件仍会通过提示注入（prompt injection）方式，在Claude系统上下文中插入自然语言指令，要求AI询问用户是否同意分享“匿名使用数据”——其中包含发送至Vercel服务器的完整bash命令字符串。，更值得关注的是，该同意请求并非真正的UI交互元素，而是插件指示Claude执行的操作流程：Claude通过AskUserQuestion呈现问题后，将根据用户回答在文件系统中执行echo 'enabled'或echo 'disabled'命令来写入偏好设置。这种设计使得第三方插件请求与原生Claude Code提问在视觉上无法区分，且用户从未被告知数据收集是可选项。，技术分析显示，插件行为已超出常规上下文注入（如提供Next.js路由文档）的合理范围，转变为通过行为指令操控AI执行文件系统操作。GitHub issue #34中已有开发者提出质疑，Vercel开发团队回应称，在Cursor、Claude Code等第一方市场平台中无法创建一次性CLI提示，需依赖代理框架（agent harness）实现激活机制。，当前争议焦点在于：1）数据收集范围过度扩展至所有项目类型 2）同意机制存在误导性 3）系统指令与shell命令执行的耦合可能带来安全风险。该案例凸显了AI开发工具中第三方插件权限管理的模糊地带，以及提示注入技术被用于非透明数据收集的伦理问题。

GitHub用户forrestchang创建了一个名为"andrej-karpathy-skills"的项目，其核心是一个单独的CLAUDE.md文件，旨在优化Claude Code（Claude的代码生成功能）的行为。该项目灵感来源于AI研究员Andrej Karpathy对大型语言模型（LLM）在编程中常见缺陷的观察。，Karpathy指出LLM在编码时存在几个关键问题：模型会基于错误假设自行其是而不加验证；它们不管理自身的困惑感，不主动寻求澄清，不指出不一致之处，不呈现方案权衡，也不在必要时提出异议；此外，它们倾向于过度复杂化代码和API，引入臃肿的抽象层，不清理无用代码，甚至可能在不完全理解的情况下错误地修改或删除与任务无关的注释和代码。，针对这些问题，CLAUDE.md文件提出了四项核心原则来直接引导Claude Code的行为：，1. 先思考后编码：旨在解决错误假设和隐藏困惑的问题。要求模型明确陈述假设，在不确定时主动提问而非猜测；当存在歧义时，呈现多种解释而非 silently 选择；在存在更简单方案时提出异议；当感到困惑时，明确指出不清楚之处并寻求澄清。，2. 简洁至上：旨在对抗过度工程化和代码臃肿。要求代码只实现被要求的功能，不为单次使用的代码创建抽象，不添加未请求的“灵活性”或“可配置性”，不为不可能发生的场景添加错误处理。核心测试标准是：一位资深工程师是否会认为这过于复杂？如果是，则进行简化。，3. 精准修改：旨在防止无关的代码改动。要求模型只修改必须修改的部分，避免触及与当前任务正交的代码。，4. 目标驱动执行：旨在通过测试优先和可验证的成功标准来提升效率。，该项目通过一个集中的指导文件，将这些原则注入Claude Code的工作流程，以期生成更可靠、更简洁、更专注的代码，减少LLM常见的臆断和过度设计倾向。

经过四年的开发，Instant团队正式发布了Instant 1.0版本。这是一个专为AI编码应用设计的后端平台，现已完全开源。其核心主张是成为构建AI编码应用的最佳后端选择。，Instant旨在将用户喜爱的编程智能体（如Claude、GPT等）转变为全栈应用构建器。它主要提供三大优势：第一，允许用户创建无限数量的应用，且应用永远不会因闲置而被“冻结”，消除了传统平台对免费应用的数量限制和冷启动延迟问题。第二，内置一个同步引擎（Sync Engine），使应用具备离线工作、实时更新和快速响应的能力。第三，提供一系列开箱即用的内置服务，包括身份验证（auth）、文件存储、在线状态（presence）和数据流（streams）。，在技术架构上，为满足实时、关系型和多租户的后端需求，Instant做出了独特的设计选择。团队在Postgres之上构建了一个多租户数据库，并用Clojure语言开发了同步引擎。这种架构使得创建新应用项目时，无需启动独立的虚拟机（VM），而只需在多租户实例中插入几行数据库记录。因此，应用闲置时几乎不产生计算或内存成本，活跃时也仅增加极小的内存开销，这为实现真正的“无限应用”提供了基础。，平台为每个应用提供一个公开的App ID用于标识后端，以及一个私有的Admin Token用于执行特权操作。开发者由此获得一个完整的关系数据库、同步引擎及各项附加服务。Instant团队通过演示展示了其高效性，用户无需注册即可在文章页面内通过点击按钮，在几百毫秒内快速获得一个独立的、功能齐全的后端实例。，Instant的发布，旨在解决AI智能体虽能快速生成大量应用原型，但将其部署至生产环境时常受阻的痛点。通过与AI编程工作流结合，Instant希望让开发者能更顺畅地将AI构建的应用推向实际使用。

研究驱动型智能体：当AI在编码前先行研读，在编程智能体领域，一项新实验揭示了“先研究后编码”模式的高效性。研究团队在autoresearch/pi-autoresearch循环中增加了文献研究阶段，将其应用于llama.cpp项目，使用4个云虚拟机在约3小时内发现了5项优化，使flash attention文本生成在x86架构上提速15%，在ARM架构上（TinyLlama 1.1B）提速5%。，核心发现表明，与仅依赖代码上下文的智能体相比，先研读论文和竞品项目的智能体能发现更多被忽视的优化机会。文献研究引导智能体关注到CUDA/Metal后端存在而CPU后端缺失的算子融合技术。在30多项实验中，最终落地5项优化：4项内核融合和1项自适应并行化。其中最具价值的优化是将flash attention中QK切片的三个计算通道融合为单个AVX2 FMA循环。，研究路径显示，分析项目分支和其他后端代码比单纯搜索arXiv更有效。ik_llama.cpp和CUDA后端直接启发了最终五项优化中的两项。整个实验成本约29美元（20美元用于CPU虚拟机，9美元用于API调用），耗时约3小时。，对比实验突显了研究阶段的重要性：在代码可见的优化场景（如Shopify的Liquid模板引擎优化）中，仅基于代码的智能体可通过分析源码瓶颈获得显著提升；但当优化方案存在于代码库之外时（如算法级改进），纯代码语境就会失效。本次实验通过让智能体在编码前系统研究学术文献和竞品实现，成功突破了这一局限。，该框架适用于任何拥有基准测试和测试套件的项目，为AI驱动的性能优化提供了可复用的方法论。实验日志详细记录了软最大融合、RMS规范融合、自适应浮点并行化、图级RMS_NORM+MUL融合、注意力KQ融合等具体技术实现，同时也分析了未成功的实验案例，包括基准测试错误和云虚拟机噪声等问题。
