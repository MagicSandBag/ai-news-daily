AI编码代理正在编写越来越多的代码，因此我们必须有意识地规范其生成的代码质量。本文档旨在为所有与AI编码代理协作的开发者提供一份宣言与指南，明确AI所写代码应遵循的标准，并可作为一项技能赋予AI代理使用（通过npx skills add theswerd/aicode添加至Cursor等工具）。，核心原则是代码应具备自解释性。如何将逻辑拆分为函数以及设计函数间传递的数据结构，直接决定代码库的长期可维护性。为此，文档提出了两种关键函数类型：，语义函数是代码库的基础构建块。其设计应尽可能精简，以确保正确性。一个良好的语义函数应接收完成目标所需的所有输入，并直接返回所有必要输出。它可以封装其他语义函数以描述特定流程，从而将复杂但通用的逻辑固化下来。语义函数通常应避免副作用（除非副作用是其明确目标），以保证在不理解内部实现的情况下可安全复用。当复杂逻辑在大型流程中功能不清晰时，最佳实践是将其分解为一系列自描述的语义函数链，每个函数仅接收所需数据、传递下一步必要结果而不做其他操作。例如quadratic_formula()或retry_with_exponential_backoff_and_run_y_in_between(x: X, y: Y)都是优秀范例。即使这些函数不再被调用，其存在也为后续开发者或AI代理提供了清晰的信息索引。语义函数应无需注释，代码本身即为其功能的定义，并且应具备高度的单元可测试性。，实用函数则用作一系列语义函数与独特逻辑的包装层，代表代码库中的复杂流程。在构建生产系统时，逻辑变得杂乱是常态，实用函数正是为此类流程提供组织框架。例如provision_new_workspace_for_github_repo(repo, user)或handle_user_signup_webhook()。它们通常不应在多个地方复用，否则应考虑将其显式逻辑拆解并移入语义函数。对实用函数的测试属于集成测试范畴，常置于整个应用功能测试的上下文中进行。这类函数的内容预计会随时间发生彻底变化。，文档强调，若缺乏有意识的规范，AI编码代理的集体输出将迅速导致代码库混乱。通过采纳上述模式，开发者可以引导AI生成更模块化、可测试且易于人类与AI协同维护的代码。
