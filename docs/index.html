微软高管提出AI代理或将需要购买软件许可证，类似企业员工。在近期一次会议上，微软执行副总裁拉杰什·贾提出了一个前瞻性观点：随着企业未来大规模部署AI代理，这些AI代理可能需要独立的身份标识，包括登录账号、收件箱以及在软件系统中的“席位”。这意味着AI代理可能被视为软件的新用户，需要为其购买许可证，即行业术语中的“席位”。，这一设想直接回应了当前企业软件行业面临的SaaS定价模式挑战。以微软、Salesforce和Workday为代表的公司长期依赖基于用户数量的“席位”定价模式。投资者担忧AI技术可能削弱该模式，因为一名员工可以管理多个AI代理，从而减少企业所需的总许可证数量。但贾认为，AI代理本身将成为新的“席位”机会：即使企业员工数量减少，每个员工配备的多个AI代理仍将产生相应的许可证需求。例如，一家公司现有20名员工购买20个Microsoft 365许可证；若未来员工减至10人，但每人配备5个AI代理，则仍需支付50个许可证费用。，然而，并非所有行业专家认同此观点。AlixPartners合伙人内纳德·米利切维奇认为，AI代理将减少人机交互需求，从而降低软件许可证总数。这一分歧凸显了AI技术对软件商业模式的核心冲击：企业需重新定义“用户”边界，软件供应商则需探索适应AI时代的定价策略。，当前，微软等公司正积极布局AI代理生态，其表态也反映了行业对AI商业化路径的探索。若AI代理需独立许可证，企业软件市场规模可能不减反增，但具体实施仍面临技术集成、成本核算和伦理规范等多重挑战。

GitHub用户forrestchang创建了一个名为"andrej-karpathy-skills"的项目，其核心是一个单独的CLAUDE.md文件，旨在改进Claude Code（Claude的代码生成功能）的行为。该项目灵感来源于AI研究员Andrej Karpathy对大型语言模型（LLM）在编码中常见缺陷的观察。，Karpathy指出LLM在编码时存在几个关键问题：模型会基于错误假设自行其是而不加验证；不善于管理困惑、寻求澄清、揭示矛盾或呈现权衡方案；在应该提出异议时保持沉默。此外，它们倾向于过度复杂化代码和API，引入臃肿的抽象层，不清理无用代码，甚至可能在不完全理解的情况下错误地修改或删除与任务无关的注释和代码。，该项目提出的解决方案浓缩为包含四大原则的CLAUDE.md文件，直接针对上述问题：，1. 先思考，后编码：旨在解决错误假设和隐藏的困惑。要求模型明确陈述假设，在不确定时主动提问而非猜测；呈现多种可能的解释，避免在存在歧义时 silently 做出选择；当存在更简单方案时提出异议；遇到困惑时明确指出不清楚之处并寻求澄清。，2. 简洁至上：对抗过度工程化的倾向。强调只编写解决当前问题所需的最少代码，不添加未要求的功能、不为单次使用的代码创建抽象、不引入未请求的“灵活性”或“可配置性”、不为不可能发生的场景添加错误处理。核心是用最少的代码解决问题，避免任何推测性设计。，3. 精准修改：防止对不相关代码进行不必要的编辑。要求更改必须严格限定在任务指定的范围内，避免产生副作用。在修改代码时，需充分理解上下文，确保不破坏现有功能，并保持代码风格一致。，4. 目标驱动执行：通过测试优先和可验证的成功标准来确保效果。鼓励在开始编写实现代码之前先定义测试或验证方法，以确保工作朝着明确、可检验的目标推进，从而更有效地利用LLM的能力。，该项目将这些原则集成到一个配置文件中，供用户在Claude Code等场景中使用，以引导AI生成更可靠、更简洁、更专注的代码，减少LLM的典型编码陷阱。项目链接提供了CLAUDE.md文件以及示例（EXAMPLES.md）供参考。
